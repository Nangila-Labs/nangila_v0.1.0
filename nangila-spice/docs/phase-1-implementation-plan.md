# Phase 1 Implementation Plan

## Objective

Phase 1 establishes the correctness foundation for Nangila v1.

The goal is to replace permissive or placeholder success conditions with a repeatable, oracle-backed validation system that proves the single-node Rust solver matches ngspice closely enough on the v1 supported subset.

Phase 1 is complete only when the official correctness gate suite in [docs/v1-simulator-contract.md](/Users/craigchirara/nangila/nangila-spice/docs/v1-simulator-contract.md) can be run automatically and produces real pass/fail results against ngspice.

## Phase 1 Outcomes

At the end of Phase 1, the project must have:

- a reproducible ngspice oracle harness
- real waveform comparison tooling
- correctness-based integration tests
- explicit failure on placeholder or synthetic execution paths
- a trusted single-node reference path for later phases

## Scope

Phase 1 covers:

- correctness validation only
- single-node Rust execution as the reference path
- the official correctness gate suite
- waveform generation, comparison, and reporting

Phase 1 does not cover:

- distributed correctness equivalence
- GPU correctness parity
- performance optimization beyond what is needed to support repeatable testing

## Correctness Gate Tiers

Phase 1 uses two correctness-gate tiers so benchmark coverage can expand without degrading the per-change developer loop.

Both tiers are part of the official Phase 1 validation program.

The distinction is operational, not contractual:

- the mandatory gate is the required per-change path
- the extended gate is the required nightly or manual path for larger cases

For Phase 1, the remaining large combinational decks are intentionally extended-only unless they later prove fast and stable enough for the mandatory CI budget.

### Mandatory Gate

The mandatory correctness gate is the required local and CI path for every production-facing change.

It should include only cases that are:

- oracle-backed against ngspice
- comfortably within the CI runtime budget
- stable across repeated runs

The mandatory Phase 1 gate currently covers:

- `simple_rc`
- `inverter`
- `sram_6t`
- `c17_full`
- `c17_synth`
- `c17_auto`
- `c432_auto`
- `s27_auto`
- `s382_auto`
- `s641_auto`

### Extended Gate

The extended correctness gate is opt-in locally and should run on a nightly or manual CI schedule.

It is for cases that are already useful and valid, but still too expensive or too noisy for the mandatory path.

The extended gate currently covers:

- `c880_auto`
- `c1355_auto`
- `c1908_auto`

Promotion from extended to mandatory should happen only when:

- repeated runs stay stable
- runtime fits the mandatory CI budget
- artifact history shows consistent headroom against the v1 tolerances

For the current Phase 1 definition, `c880_auto`, `c1355_auto`, and `c1908_auto` are official validation cases but are intentionally not part of the per-change gate.

## Workstreams

### Workstream 1: Freeze the Single-Node Reference Path

Objective:

- make one runtime path authoritative for correctness work

Tasks:

- declare the single-node Rust solver as the only Phase 1 reference engine
- ensure production-facing correctness tests do not depend on Python synthetic fallback paths
- prevent end-to-end correctness runs from passing if solver output is:
  - empty
  - all-zero by accident
  - a single final snapshot when a waveform is expected
  - generated by synthetic placeholder logic

Deliverables:

- documented reference execution command
- explicit test assertions rejecting placeholder behavior

Exit criteria:

- every correctness-gated test runs through the single-node Rust path or fails

### Workstream 2: Build the ngspice Oracle Harness

Objective:

- create a deterministic way to compare Nangila outputs against ngspice

Tasks:

- define a standard transient run configuration format:
  - netlist path
  - stop time
  - timestep
  - selected output nodes
- build a runner that:
  - executes ngspice on a benchmark netlist
  - captures waveform output in a stable machine-readable format
  - executes Nangila with matching runtime parameters
  - captures Nangila waveform output in the same normalized schema
- normalize:
  - node naming
  - time axes
  - units
  - missing-node handling

Deliverables:

- oracle runner script or test utility
- common waveform artifact format
- benchmark run manifest format

Exit criteria:

- the same benchmark can be run through both tools and produce comparable waveform artifacts automatically

### Workstream 3: Build the Waveform Comparison Engine

Objective:

- turn waveform comparison into explicit pass/fail metrics

Tasks:

- implement comparison metrics from the v1 simulator contract:
  - maximum absolute node-voltage error
  - RMS node-voltage error
  - final-time node-voltage error
  - 50% VDD edge timing error where applicable
- add interpolation or aligned sampling logic so mismatched timestep grids can still be compared fairly
- produce structured comparison reports for each benchmark

Deliverables:

- waveform comparator module
- machine-readable result schema
- human-readable correctness summary report

Exit criteria:

- each benchmark run produces explicit metric values and a pass/fail result against the contract tolerances

### Workstream 4: Replace Weak Integration Tests

Objective:

- make the Python and Rust integration suites enforce numerical correctness instead of basic execution

Tasks:

- replace file-exists and non-empty assertions with waveform assertions
- add benchmark-specific expectations for:
  - RC charging shape
  - inverter switching behavior
  - SRAM stable states
  - synthesized logic outputs on selected nodes
- fail tests when:
  - point count is too low
  - expected nodes are missing
  - waveforms are flat when they should switch
  - voltage ranges are nonphysical

Deliverables:

- rewritten end-to-end correctness tests
- benchmark-specific assertions for the correctness gate suite

Exit criteria:

- obviously wrong outputs fail immediately in CI and local test runs

### Workstream 5: Standardize Waveform Output

Objective:

- make Nangila emit artifacts suitable for correctness verification

Tasks:

- define the production waveform output contract:
  - full time series
  - node names
  - metadata
  - run configuration
- ensure the single-node solver emits full waveforms for correctness-gated runs
- stop reducing solver output to one-point summaries in correctness workflows

Deliverables:

- stable waveform output schema
- CLI or test mode that emits correctness-grade artifacts

Exit criteria:

- correctness runs always produce full waveforms that can be fed into the comparator

### Workstream 6: Define and Automate the Correctness Gate Suite

Objective:

- turn the benchmark list into an executable release gate

Tasks:

- codify the official correctness gate suite from the v1 contract:
  - `tests/spice_examples/simple_rc.sp`
  - `benchmarks/reference_circuits/inverter.sp`
  - `benchmarks/reference_circuits/sram_6t.sp`
  - `benchmarks/reference_circuits/c17_synth.sp`
  - `benchmarks/reference_circuits/c17_full.sp`
  - selected `benchmarks/auto_synth/*.sp` circuits from the contract
- define runtime configurations for each benchmark:
  - stop time
  - timestep
  - monitored nodes
  - expected switching events
- add one command that runs the full correctness gate and produces a summary

Deliverables:

- benchmark manifest for the correctness gate suite
- single command or script for full correctness validation

Exit criteria:

- the mandatory correctness gate can be run locally and in CI through one documented entrypoint
- the extended correctness gate can be run locally and on a nightly or manual CI schedule through one documented entrypoint

### Workstream 7: Add Failure Diagnostics

Objective:

- make correctness failures actionable rather than opaque

Tasks:

- report per-benchmark failure reason categories:
  - parser failure
  - solver failure
  - non-convergence
  - missing waveform
  - tolerance violation
  - node mismatch
- capture artifact bundles for failed runs:
  - netlist name
  - run config
  - ngspice waveform
  - Nangila waveform
  - comparison summary
  - solver logs

Deliverables:

- failure report schema
- artifact directory layout for failed correctness runs

Exit criteria:

- a failed correctness run gives enough information to debug the defect without rerunning blindly

## Suggested Execution Order

1. Freeze the single-node reference path.
2. Standardize Nangila waveform output.
3. Build the ngspice oracle runner.
4. Build the waveform comparator.
5. Replace weak integration tests.
6. Encode the official correctness gate suite.
7. Add reporting and failure diagnostics.

## Milestones

### Milestone 1: Real Waveforms Only

Success criteria:

- correctness runs emit full Nangila waveforms
- placeholder outputs are rejected
- single-node execution is the only accepted correctness path

### Milestone 2: Oracle Comparison Running

Success criteria:

- ngspice and Nangila outputs can be produced and normalized for the same benchmark
- comparison metrics are computed automatically

### Milestone 3: Correctness Gate Enforced

Success criteria:

- official correctness suite runs through one command
- pass/fail decisions are based on contract tolerances
- CI can run the gate automatically

## Deliverables

Phase 1 should produce at least the following concrete artifacts:

- oracle runner tooling
- waveform comparison tooling
- benchmark manifest
- rewritten correctness-focused integration tests
- structured correctness reports
- documented correctness command for local and CI use

## Exit Criteria

Phase 1 is complete only if all of the following are true:

- the official correctness gate suite can be executed automatically
- ngspice is integrated as the correctness oracle
- Nangila output is compared using the contract metrics
- placeholder or synthetic execution paths cannot pass correctness tests
- the single-node Rust solver is established as the trusted reference path for later phases

## Recommended First Implementation Slice

The first practical slice of Phase 1 should be:

1. make the single-node solver emit full waveforms for `simple_rc`, `inverter`, and `sram_6t`
2. add an ngspice runner for those same three benchmarks
3. implement waveform comparison with max error, RMS error, and final-value error
4. rewrite the existing end-to-end tests to fail on bad waveforms

That slice is small enough to finish, but strong enough to expose the real correctness gaps immediately.
