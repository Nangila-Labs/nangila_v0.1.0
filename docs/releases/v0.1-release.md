# Nangila v0.1 Release Criteria

This document defines what must be true before calling the repository `v0.1`-ready.

## Required Gates

### Build

- A fresh clone can create a virtual environment.
- `maturin develop --release -F python` succeeds on the supported source-install path.
- Importing `nangila` does not require CUDA.

### Test

- `cargo test` passes for the workspace.
- `python -m pytest -q` passes in the local editable-install path.
- Experimental integration scripts are excluded from the default CI gate.
- `bash scripts/cuda_single_gpu_smoke.sh` passes on one supported NVIDIA CUDA host.
- `CUDA_VISIBLE_DEVICES=0,1 python -m torch.distributed.run --standalone --nproc_per_node=2 tests/integration/test_ddp_correctness.py` passes on one supported Linux + CUDA setup.

### Docs

- README describes the actual supported surface, not the long-term vision.
- The support matrix is explicit.
- Experimental APIs are labeled as experimental.
- The release checklist and changelog template are available for the release pass.

### CI

- CI runs the Rust workspace test suite.
- CI runs the Python editable-install smoke path in a virtual environment.
- The CI baseline reflects the commands documented in the README.

## Non-Goals for v0.1

- Production-grade FSDP support
- Guaranteed GPU-native Python fast-path qualification
- Multi-node scaling claims
- Universal framework positioning

## Expected Outcome

If these gates are met, `v0.1` is a narrow but credible product:
- a Rust-backed DDP gradient compression package,
- installable from source,
- with a clear support boundary,
- and a reproducible green baseline.

The repository-level release boundary for this version is also recorded in
[`nangila_v01/`](/Users/craigchirara/nangila/nangila_v01/), so `v0.1` is
explicitly defined without forking the implementation into a second source tree.
